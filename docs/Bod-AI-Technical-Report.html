1. Executive summary

Bod AI is a full-stack application that simulates a corporate board deliberation on arbitrary business scenarios. Four LLM-backed executive personas (CEO, CFO, CMO, Chief Risk Officer) each produce an initial position, engage in configurable multi-round debate, and converge on a collective verdict with confidence, rationale, supporting arguments, and key disagreements. The user interface streams results in real time via Server-Sent Events (SSE), visualizes agent activity in a Three.js voxel office, and supports optional news context injection, session persistence, human-in-the-loop comments (post-session board responses), memo export, replay, and session history.

2. Objectives & learning outcomes

  • Demonstrate multi-agent orchestration with role-specific prompts and structured JSON outputs.
  • Implement streaming UX so users observe progressive reasoning rather than a single blocking response.
  • Provide explainability: weighted or majority aggregation plus an LLM-generated narrative summary.
  • Integrate a modern web stack (FastAPI + Next.js) with 3D visualization and printable reporting.

3. System architecture

The system follows a client–server pattern. The browser runs the Next.js SPA; the Python backend calls OpenAI and optional news APIs.

3.1 Simulation pipeline (high level)

  1. News (optional): Scenario text is used to fetch relevant headlines when NEWS_API_KEY is set.
  2. Phase 1 — Initial positions: Each agent returns stance (approve / conditional / reject), reasoning, and key concern.
  3. Phase 2 — Debate rounds: For each round, every agent responds to peers with a new argument and optional stance update; prior arguments are accumulated in memory.
  4. Phase 3 — Decision: DecisionEngine applies weighted scoring or majority vote, then calls the LLM to synthesize the final structured summary.
  5. Persist: Full state is saved under backend/logs/<session_id>.json.

3.2 Human-in-the-loop extension

After a run completes, users may submit a comment in the live discussion panel. The backend exposes POST /comment/stream, which streams one response per executive directed at the stakeholder, using the same debate context and the new user utterance.

5.3 Decision logic

  • Weighted mode: Maps stances to scores (approve=1.0, conditional=0.5, reject=0.0), multiplies by per-agent weights (default 25% each), derives confidence and verdict bands (e.g. Approved vs Conditional vs Rejected).
  • Majority mode: Counts final stances and picks the modal category.
  • Summary: A follow-up completion produces structured JSON: supporting arguments, disagreements, and human-readable rationale referencing named executives and vote weights.

5.4 Configuration & security notes

  • OPENAI_API_KEY loaded from environment or backend/.env (must not be committed).
  • Optional NEWS_API_KEY for news augmentation.
  • CORS allows configured local/LAN origins for development.

6. Frontend design

  • app/page.tsx: Main boardroom UI — scenario input, weight sliders (must total 100%), debate rounds, weighted vs majority mode, SSE event reducer mapping to chat messages and agent states.
  • components/Office3D.tsx: Three.js scene; agents highlight when “speaking.”
  • components/ConfidenceMeter.tsx: Live weighted stance visualization.
  • components/ExportPDF.tsx: Printable / downloadable HTML memo with scenario, verdict, rationale, weights used, transcript summary.
  • lib/api.ts: Typed SSE client for /simulate/stream and /comment/stream.
  • Additional UX: History sidebar, replay timeline, compare view, stance-change toasts.

7. Data & persistence

Each completed simulation is stored as a JSON document in backend/logs/ containing scenario, initial positions, debate rounds, final decision metadata, and timestamps. This supports auditing, reloading sessions in the UI, and offline review.

8. Testing & operations

  • Backend: uvicorn main:app --reload --port 8000; interactive docs at /docs.
  • Frontend: npm run dev (default port 3000).
  • Typical run latency: on the order of tens of seconds depending on rounds, model, and network.

9. Limitations & future work

  • Simulation quality depends on the LLM and prompt design; outputs are not financial or legal advice.
  • News retrieval is optional and API-dependent.
  • Possible extensions: authentication, multi-tenant storage, richer evaluation metrics, agent tool use (e.g. spreadsheets), formal test suites, CI/CD.

10. Conclusion

Bod AI demonstrates a complete multi-agent deliberation system with streaming UX, structured aggregation, and explainable summaries, implemented with industry-standard web and AI tooling suitable for coursework demonstration and further research.
